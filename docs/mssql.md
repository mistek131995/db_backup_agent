## Логический бэкап

Логический бэкап снимает **одну базу данных** через `Microsoft.SqlServer.DacFx` — in-process, без внешних бинарников. Агент может находиться в любом месте сети — доступ к файловой системе SQL Server не нужен ни для бэкапа, ни для восстановления.

### Бэкап

`DacServices.ExportBacpac` подключается к SQL Server по TDS-протоколу и выгружает схему и данные выбранной базы в формат `.bacpac` (ZIP-архив). Агент сохраняет файл в `OutputPath` и прогоняет через стандартный pipeline: шифрование AES-256-GCM → загрузка в S3/SFTP.

**Требования**

- TDS-доступ к SQL Server (порт 1433 по умолчанию).
- Пользователь в `ConnectionConfig` имеет права на подключение к базе и чтение всех схем/таблиц, которые нужно сохранить. Минимум:

  ```sql
  CREATE LOGIN backup_user WITH PASSWORD = '...';
  USE mydb;
  CREATE USER backup_user FOR LOGIN backup_user;
  ALTER ROLE db_datareader ADD MEMBER backup_user;
  GRANT VIEW DEFINITION TO backup_user;
  GRANT VIEW DATABASE STATE TO backup_user;
  ```

### Восстановление

Восстановление выполняется через `DacServices.ImportBacpac` по TDS-соединению — SQL Server не останавливается, остальные базы на сервере не затрагиваются.

Агент последовательно:

1. Скачивает и расшифровывает `.bacpac` во временный файл на диске.
2. Удаляет целевую базу (если существует) через T-SQL: `DROP DATABASE IF EXISTS <target>`.
3. Разворачивает `.bacpac`: `DacServices.ImportBacpac`.

**Требования**

- TDS-доступ к SQL Server.
- Пользователь в `ConnectionConfig` имеет привилегию `dbcreator` на уровне сервера (для создания базы):

  ```sql
  ALTER SERVER ROLE dbcreator ADD MEMBER backup_user;
  ```

**Что нужно знать**

- Целевая база **всегда пересоздаётся**: все существующие данные в ней уничтожаются.
- Можно восстановить в базу с **другим именем** или на **другой сервер**.
- Восстановление медленнее физического на больших базах: данные импортируются построчно, индексы перестраиваются.

### Трудности и ограничения

Логический бэкап **не делает мгновенный снимок** базы. Выгрузка идёт объект за объектом, и изменения, произошедшие за это время, могут попасть в файл в несогласованном виде. Если важна точная копия на конкретный момент — используйте физический режим.

Режим не справится, если в базе встречается что-то из перечисленного:

- **Ссылки на другие базы** — процедуры, вьюхи или триггеры, обращающиеся к таблицам соседней БД на том же сервере. Для логического бэкапа база должна быть самодостаточной; такие ссылки валят экспорт, и обойти это нельзя.
- **Ссылки на другие серверы** — linked servers.
- **«Осиротевшие» пользователи** — учётки в базе, у которых на сервере уже нет соответствующего логина.
- **CLR-сборки** с повышенными разрешениями (`UNSAFE`, `EXTERNAL_ACCESS`).
- **Service Broker** с очередями или диалогами, уходящими за пределы базы.
- **Change Data Capture / Change Tracking** — поддерживаются частично.
- **Репликация** — настройки publisher/subscriber в бэкап не попадут.
- **FileStream / FileTable** — ограниченная совместимость.
- **TDE и внешние провайдеры ключей** — ключи шифрования уровня БД не переносятся.
- **Темпоральные таблицы** с историей в отдельной таблице.
- **Объекты уровня сервера** — логины, роли сервера, задачи SQL Agent, linked servers. Их в этом режиме нет в принципе, это не бэкап сервера.

## Физический бэкап

Физический бэкап использует штатный механизм SQL Server: `BACKUP DATABASE ... TO DISK`. Файл `.bak` появляется на диске хоста SQL Server — агент не может получить его иначе как через **общий каталог**, видимый обеим сторонам.

### Бэкап

Агент выполняет `BACKUP DATABASE ... TO DISK = '<SharedBackupPath>/...'` через TDS-соединение. SQL Server записывает `.bak` в указанный каталог. Затем агент читает файл по пути `AgentBackupPath` (или `SharedBackupPath`, если они совпадают) и прогоняет через стандартный pipeline: шифрование AES-256-GCM → загрузка в S3/SFTP. Временный `.bak` удаляется.

**Общий каталог**

Агенту и SQL Server нужен **один физический каталог**, доступный обеим сторонам. Путь к нему с каждой из сторон может быть разным — для этого в конфиге два поля:

- `SharedBackupPath` — путь так, как его видит **SQL Server**. Уходит в команду `BACKUP`/`RESTORE`.
- `AgentBackupPath` — путь к **той же папке** так, как её видит **агент**. Если пути совпадают — оставить пустым.

**Если `SharedBackupPath` не задан**, агент запрашивает у SQL Server `SERVERPROPERTY('InstanceDefaultBackupPath')` и использует его (на Windows по умолчанию это `C:\Program Files\Microsoft SQL Server\MSSQL<ver>.<instance>\MSSQL\Backup\`). Сценарий рассчитан на то, что агент и SQL Server работают на одном хосте и оба видят этот каталог одинаково. На remote SQL без `SharedBackupPath` бэкап упадёт с понятной ошибкой — агент физически не сможет дотянуться до локальной для SQL папки.

| Сценарий | `SharedBackupPath` | `AgentBackupPath` |
|---|---|---|
| Агент и SQL Server на одном хосте (без Docker) | *(не задано — возьмётся `InstanceDefaultBackupPath`)* | *(не задано)* |
| Одинаковый хост, но другой каталог для `.bak` | `C:\mssql-backups` | *(не задано)* |
| Docker Compose, общий named volume у обоих контейнеров | `/var/opt/mssql/backups` | *(не задано)* |
| Агент в Docker, SQL Server на хосте; bind-mount одинакового пути | `/var/opt/mssql/backups` | *(не задано)* |
| Windows SQL Server + Linux-контейнер агента, SMB/UNC-шара смонтирована в контейнер | `\\fs\backup` | `/mnt/backup` |
| Оба на разных хостах, NFS-шара | `/mnt/nfs/mssql-backups` | *(не задано)* |

Агент **не** настраивает volume-ы, SMB/CIFS-монтирование или NFS — он только читает и пишет по указанным путям. Проверьте права на запись/чтение с обеих сторон до попытки бэкапа.

> `DatabaseConfig.OutputPath` для MSSQL physical **не используется**. Временным/общим каталогом всегда управляет пара `SharedBackupPath`/`AgentBackupPath` или `InstanceDefaultBackupPath` из SQL Server.

**Требования**

- SQL Server должен иметь право **писать** в `SharedBackupPath` (для backup) и **читать** оттуда (для restore).
- Агент должен иметь право **читать** из `AgentBackupPath` (для upload) и **писать** туда (для restore — он кладёт туда расшифрованный `.bak` перед `RESTORE`).
- TDS-доступ к SQL Server.
- Пользователь в `ConnectionConfig` имеет роль `db_backupoperator` или является `sysadmin`:

  ```sql
  ALTER SERVER ROLE db_backupoperator ADD MEMBER backup_user;
  ```

### Восстановление

Агент последовательно:

1. Скачивает и расшифровывает `*.bak` в каталог `AgentBackupPath`.
2. Выполняет `RESTORE DATABASE <target> FROM DISK = '<SharedBackupPath>/...' WITH REPLACE` через TDS-соединение.
3. Удаляет временный `.bak` из каталога.

**Требования**

- Тот же общий каталог, что и для бэкапа (см. выше).
- Пользователь в `ConnectionConfig` имеет роль `sysadmin` или `dbcreator` + `db_backupoperator` на уровне сервера.

**Сценарии развёртывания**

| Сценарий | Бэкап | Восстановление |
|---|---|---|
| Агент и SQL Server на одном хосте (bare-metal / VM) | ✅ | ✅ |
| Docker Compose: общий named volume | ✅ | ✅ |
| Агент в Docker, SQL Server на хосте; bind-mount | ✅ | ✅ |
| Windows SQL Server + Linux-контейнер агента; SMB-шара | ✅ | ✅ |
| Агент и SQL Server на разных хостах без общего каталога | ❌ | ❌ |

### Трудности и ограничения

**Общий каталог обязателен для remote SQL Server.**
Без `SharedBackupPath` агент не может передать `.bak` между собой и SQL Server: при backup SQL Server пишет файл в локальный для себя каталог, который агент не видит; при restore агент расшифровывает файл у себя, но `RESTORE DATABASE ... FROM DISK = ...` выполняется на стороне SQL Server, который не видит файловую систему агента. При попытке выполнить backup или restore на remote SQL Server без `SharedBackupPath` агент явно падает с ошибкой и подробным сообщением о том, что именно задать в конфиге.

**Восстановление разрушает данные целевой базы.**
`RESTORE ... WITH REPLACE` перезаписывает существующую базу. Активные соединения к ней разрываются SQL Server автоматически. Отменить операцию после начала `RESTORE` невозможно без отдельной копии.

**Восстановление на более старую версию невозможно.**
Файл `.bak` привязан к версии SQL Server — восстановить его на более старой версии сервер откажется. Для миграции на другую версию используйте логический режим.

**Ответственность за монтирование — на пользователе.**
Агент не настраивает volume-ы, SMB/CIFS или NFS. Выбирайте то, что подходит вашей инфраструктуре: Docker named volume (если оба контейнера в одном compose), bind-mount (если один из них на хосте), SMB/CIFS (Windows SQL Server + Linux-агент), NFS (кросс-хост на Linux).

**FILESTREAM filegroup не поддерживается.**
Если в БД есть FILESTREAM-файлы (тип `S` в `sys.database_files`), агент откажет в бэкапе на этапе preflight — накапливать `.bak`, который потом не восстановить, бессмысленно. Симметрично, restore старого `.bak` с FILESTREAM завершится явной ошибкой. Поддержка требует доступа к файловой системе SQL Server (для очистки orphan-каталогов от прерванных restore) и в текущей версии не реализована.
